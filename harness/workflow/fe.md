# 프론트엔드 / 3D 클라이언트 구현 워크플로우

> 대상: `client/` (React + React Three Fiber + Three.js + TypeScript + Zustand + Vite).
> 관련 ADR: ADR-0001(클라이언트 스택)·ADR-0003(네트코드, 예측/보간의 서버 경계)·
> ADR-0007(맵 파이프라인). 관련 RQ: RQ-50~55(HUD), RQ-42(스프레이), RQ-70/71(효과),
> RQ-40(채팅 입력 차단).

## 레이어 분리 규칙

클라이언트는 4개 레이어로 나뉘고, 데이터는 한 방향으로만 흐른다:

```
netcode (Colyseus 수신·예측·보간) → game state (Zustand store) → scene (R3F) → HUD (DOM)
```

| 레이어 | 책임 | 위치 | 비고 |
|---|---|---|---|
| netcode | 서버 스냅샷 수신, 로컬 입력 예측, 다른 플레이어 보간 (RQ-62/63) | `src/net/` | Colyseus 룸 상태를 구독하고 game state로 반영. 서버 상태 위에 로컬 예측을 얹되, 서버 값이 도착하면 조정한다 (RQ-62). |
| game state | 정규화된 게임 상태 저장소 | `src/store/` (Zustand) | 진실 공급원은 서버(RQ-61) — 이 store는 캐시일 뿐, 클라이언트가 새 진실을 만들지 않는다. |
| scene | R3F 컴포넌트 트리, 3D 렌더링 | `src/scene/` | 캔버스 안. 매 프레임 실행되는 코드가 있다 — 아래 프레임 예산 규칙 적용. |
| HUD | 일반 React DOM | `src/hud/` | 캔버스 밖. RQ-50~55. |

**HUD는 R3F 캔버스 밖의 DOM으로 만든다 (RQ-50~55).** 이유: 텍스트 렌더링
품질(서브픽셀 안티에일리어싱)·접근성(스크린리더·시맨틱 마크업)·리렌더 비용
(DOM diffing이 WebGL 드로우콜보다 싸다) 세 가지 모두 DOM이 유리하다. 미니맵처럼
3D 좌표를 그려야 하는 요소도 2D canvas나 SVG로 별도 구현하고 R3F 씬에 얹지 않는다.

**R3F 컴포넌트 안에서 Zustand를 `useStore()` 구독으로 읽지 않는다.** 이건 R3F에서
가장 흔한 성능 실수다 — 구독한 컴포넌트가 store 값이 바뀔 때마다 React 리렌더를
일으키고, 게임 상태는 매 틱(30Hz) 갱신되므로 사실상 매 프레임 리렌더가 걸린다.
대신 `useFrame` 콜백 안에서 `store.getState()`로 직접 읽거나, transient
subscription(`store.subscribe`)으로 구독해 `useFrame` 안에서 Three.js 객체의
속성을 직접 갱신한다. React 리렌더 경로를 타지 않는다.

```
// 금지 — 매 프레임(30Hz 갱신) 리렌더
const position = useStore((s) => s.players[id].position);

// 허용 — useFrame 안에서 직접 읽고 mesh를 직접 갱신
useFrame(() => {
  const position = useStore.getState().players[id].position;
  meshRef.current.position.set(position.x, position.y, position.z);
});
```

HUD 레이어는 이 규칙의 예외다 — HUD는 캔버스 밖 일반 DOM이므로 React
리렌더 비용이 R3F 프레임 예산에 들어가지 않는다. `useStore()` 구독을 그대로 쓴다.

## 프레임 예산

60fps = 16.6ms/frame. `useFrame`과 그 안에서 호출되는 모든 코드가 이 예산
안에 있어야 한다.

- **렌더 루프에서 할당(allocation)을 금지한다.** `new THREE.Vector3()`,
  `new THREE.Quaternion()` 등을 `useFrame` 안에서 매 프레임 생성하지 않는다.
  컴포넌트 초기화 시점(모듈 스코프 또는 `useRef`/`useMemo`)에 만들어두고
  `useFrame` 안에서는 `.set()`/`.copy()`로 값만 갱신한다.
- 배열·객체 리터럴도 동일 — `useFrame` 안에서 `{}`, `[]`를 새로 만들면 GC
  압박이 걸려 프레임 드랍(스터터)으로 이어진다.
- 리뷰 게이트(`review-gate.md`)가 이 위반을 major~blocker로 검토한다.

## 입력 처리

- **포인터 락**: FPS 시점 조작은 Pointer Lock API로 마우스를 캡처한다.
  캔버스 클릭 시 요청, ESC/포커스 이탈 시 해제.
- **채팅 입력 중 게임 입력 차단 (RQ-40)**: 채팅 입력창이 포커스를 가지면
  이동·사격 키 입력을 게임 레이어로 전달하지 않는다. 포커스 상태를 game
  state(또는 별도 UI 상태)에 두고 입력 핸들러 최상단에서 게이트한다 —
  개별 핸들러마다 분산 체크하지 않는다 (누락 위험).
- **키 바인딩 소스**: 재바인딩 UI는 스펙에 없다 (요구사항 미기재) — 클라이언트
  상수 파일(`src/input/keymap.ts`)에 고정 바인딩으로 정의한다. 사용자 설정
  가능한 리바인딩이 필요해지면 별도 RQ·ADR 대상.

## 에셋 로딩

- **glTF 맵** (ADR-0007 맵 파이프라인): 맵 지오메트리는 단일 glTF로 로드.
  로딩 완료 전 씬을 렌더링하지 않는다.
- **스프레이 비트맵 100종 (RQ-42)**: 100개 전부를 시작 시 프리로드할지,
  최초 사용 시 지연 로드할지는 ADR-0007 또는 별도 결정 필요 — 🟡. 프리로드
  시 텍스처 아틀라스로 묶어 드로우콜을 줄인다.
- **텍스처 아틀라스**: 반복 사용되는 소재(탄흔·혈흔·스프레이)는 개별
  텍스처 대신 아틀라스로 묶어 배칭한다.
- **Suspense 경계**: R3F 씬 전체를 `<Suspense>`로 감싸고, 로딩 중에는 HUD
  레이어(캔버스 밖)에서 로딩 화면을 보여준다 — 로딩 화면 자체는 R3F 컴포넌트가
  아니다.

## 효과 처리

**탄흔(RQ-70)·혈흔(RQ-71)은 인스턴싱 + 링버퍼 상한으로 구현한다. 개별 mesh
생성을 금지한다.** 명중마다 `new THREE.Mesh(...)`를 생성하면 드로우콜과 GC
압박이 명중 빈도에 비례해 무한히 늘어난다. `InstancedMesh`에 고정 상한(예:
탄흔 200개, 혈흔 100개)을 두고, 상한 도달 시 가장 오래된 인스턴스의 슬롯을
재사용하는 링버퍼로 관리한다. 상한·제거 순서는 RQ-70이 명시한 "개수 상한을
두고 오래된 것부터 제거"를 그대로 따른다.

## 디자인 진실 공급원

`docs/design/DESIGN.md`는 **아직 없다 — 🟡**. UI(HUD·크로스헤어·킬 피드 등)
작업에 착수하기 전에 이 문서가 있어야 한다 (레이아웃은 requirements.md
§6가 위치만 정하고 시각 디자인은 정하지 않았다). 없는 상태로 진행하면
디자인 드리프트가 구현 이후에 발견된다.

## 검증

FE 변경은 이 프로젝트의 TDD 파이프라인(`tdd.md`) 단위 테스트 대상이 아니다
(렌더링은 "무엇을 어떤 레벨로 테스트하는가" 표에서 테스트 대상 제외). 대신
아래가 게이트다:

| 게이트 | 확인 대상 |
|---|---|
| 타입체크 (`tsc --noEmit`) | 타입 오류 없음 |
| lint | 스타일·정적 규칙 위반 없음 |
| 스모크 | 앱 부팅 후 크래시 없음, 캔버스 마운트 확인 (자동화 가능) |
| 스크린샷 수동 확인 | 레이아웃·스타일 회귀 — 자동화 대상 아님, 사람이 확인 |

review-gate.md의 리뷰 항목("렌더 루프 내 할당")이 성능 관련 코드 리뷰를
겸한다 — FE 변경에 대해 별도 성능 테스트 게이트는 두지 않는다.
