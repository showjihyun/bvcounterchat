# ADR-0010: 프로젝트 레이아웃 — 단일 패키지 + `src/{client,server,shared}`

- 상태: 승인
- 날짜: 2026-07-21
- 관련 스펙: RQ-01, RQ-62, RQ-80 / 관련 ADR: [ADR-0001](0001-client-rendering-stack.md), [ADR-0002](0002-realtime-transport.md), [ADR-0003](0003-netcode-authority.md), [ADR-0008](0008-test-strategy.md), [ADR-0009](0009-deployment.md)

## 맥락

로드맵 1단계(프로젝트 초기화) 착수 시점에 디렉토리 레이아웃을 정해야 한다.
ADR-0001~0009 어디에도 이 결정이 없었고, 레이아웃은 세 가지를 동시에 규정한다:

1. **공유 코드의 성립 여부.** RQ-62(클라이언트 예측)와 ADR-0003은 클라이언트가
   서버와 **동일한 시뮬레이션**을 돌려 예측하고, 서버 스냅샷이 오면 재조정할
   것을 요구한다. 틱 함수·무기 수치(RQ-90)·이동 파라미터(RQ-92)가 양쪽에서
   갈리면 예측이 구조적으로 빗나간다. 공유 코드는 편의가 아니라 정확성의 전제다.
2. **빌드 토폴로지.** ADR-0009는 정적 클라이언트와 Colyseus 서버를 **하나의
   Docker 이미지**로 배포한다. 레이아웃이 빌드 산출물의 복사 경로를 정한다.
3. **스펙 동결 게이트의 차단 범위.** `.claude/hooks/gate_spec_freeze.py`의
   `BLOCKED_TOP_DIRS`가 실제 레이아웃과 어긋나면 게이트에 구멍이 생긴다.

## 결정

**단일 `package.json` + `src/` 하위 분할.**

```
src/
  client/   React + R3F. Vite가 번들. 브라우저에서만 실행.
  server/   Fastify + Colyseus. Node에서만 실행.
  shared/   양쪽이 임포트. 틱 시뮬레이션·상수·타입·Colyseus 스키마.
tests/
  unit/         순수 로직 (탄도·데미지·이동 계산)
  integration/  Colyseus 경계 (룸 동작·네트코드)
```

- 의존성·스크립트·lint·typecheck 설정은 루트 1벌로 유지한다.
- `shared`는 `@shared/*` 경로 별칭으로 임포트한다(tsconfig `paths` +
  Vite `resolve.alias` 양쪽에 동일하게 선언 — 한쪽만 선언하면 타입은 통과하고
  번들이 깨진다).
- `src/shared`는 **런타임 환경에 의존하는 API를 임포트하지 않는다.** `window`·
  `document`(브라우저 전용)도, `fs`·`process`(Node 전용)도 금지다. 양쪽에서
  같은 코드가 돌아야 하므로 이 규칙이 깨지면 공유가 성립하지 않는다.

## 근거

- **10인 규모·1인 개발·단일 이미지 배포**라는 조건에서 패키지 경계를 나누는
  비용이 이득을 넘는다. 경계가 사는 상황은 독립 버저닝·독립 배포·다수
  소비자인데, 이 프로젝트엔 셋 다 없다.
- 경로 별칭 하나로 공유가 성립하고, `tsc --noEmit`가 클라이언트·서버·공유를
  **한 번의 프로그램 그래프**로 검사한다 — ADR-0008이 요구하는 전역 타입
  검증과 정합한다(패키지를 나누면 프로젝트 참조 설정이 추가로 필요하다).
- 버린 대안:
  - **npm workspaces 모노레포**(`packages/{client,server,shared}`): 의존성
    경계는 명확해지지만 빌드 순서·호이스팅·Docker 레이어 복사 경로가 복잡해진다.
    1인 개발에서 이 복잡도는 순손실이고, `shared`를 진짜 패키지로 만들 이유
    (외부 소비자)가 없다.
  - **루트에 `client/`·`server/`·`shared/` 각자 package.json**: 워크스페이스
    없이 디렉토리만 나누면 `shared` 공유가 상대경로(`../shared`)나 수작업
    링크가 되어 가장 깨지기 쉽다. 번들러·tsc·테스트 러너가 각각 다르게 해석한다.
  - **클라이언트/서버 저장소 분리**: 공유 코드가 별도 배포 단위가 되어 버전
    스큐가 생긴다. 예측 정확성이 버전 일치에 의존하게 되므로 RQ-62에 해롭다.

## 결과

- **클라이언트와 서버 의존성이 한 `package.json`에 섞인다.** `three`와
  `colyseus`가 같은 `dependencies`에 놓인다. 번들 산출물에는 영향이 없다
  (Vite가 클라이언트 진입점에서 도달 가능한 것만 번들). 다만 `npm ls`로
  "이건 누구 의존성인가"를 구분할 수 없고, 서버 프로덕션 이미지에 클라이언트
  빌드 전용 패키지가 함께 설치된다 — 이미지 크기 손해를 감수한다.
- **`src/shared`의 환경 중립성은 도구가 강제하지 않는다.** 누군가 거기서
  `window`를 참조하면 서버에서 런타임에 터진다. 규칙이 문서에만 있는 상태이며,
  반복되면 lint 규칙(`no-restricted-imports`/`no-restricted-globals`)으로
  승격한다 — `harness/sensor-catalog.md`의 "같은 실수 2회면 센서 추가" 원칙.
- **게이트의 `BLOCKED_TOP_DIRS`는 이 결정에 종속된다.** 이 ADR을 대체하는
  레이아웃 변경은 반드시 게이트 갱신을 동반해야 한다. 빠뜨리면 게이트는
  조용히 통과시키기 시작하고, 조용히 고장 난 센서가 가장 위험하다.
- 나중에 규모가 커져 워크스페이스로 옮기려면 `src/*` → `packages/*` 이동과
  경로 별칭 제거가 필요하다 — 되돌리기 비용은 중간 수준이며, 임포트가 전부
  별칭을 거치므로 일괄 치환이 가능하다.
