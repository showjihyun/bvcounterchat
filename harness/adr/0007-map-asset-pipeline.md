# ADR-0007: 맵 에셋 파이프라인 — 오리지널 Dust2류 맵, glTF, 충돌/시각 메시 분리

- 상태: 승인 (근거: RQ-30~32, RQ-42, RQ-94 확정 — 2026-07-21 Deep Interview)
- 날짜: 2026-07-21
- 관련 스펙: RQ-30, RQ-31, RQ-32, RQ-21, RQ-42, RQ-94

## 맥락

RQ-30이 Dust2 분위기의 **오리지널** 맵(원본 저작권 회피)을, RQ-31이
스폰=Safe Zone 무효화를, RQ-21이 사다리 볼륨을 요구한다. req는 애초
저작권 제약과 대략적 분위기만 정했고, 에셋 저작 포맷·충돌 메시와 시각
메시의 분리 여부·특수 볼륨(스폰/세이프존/사다리)을 데이터로 어떻게
표현할지, 맵의 정확한 규모, 스프레이(RQ-42)의 로딩 전략은 침묵했다. Deep
Interview(질문 14~19·21)로 RQ-30~32(맵 규모·배치)와 RQ-94(스프레이
출처)가 확정되어 이 ADR을 승인으로 전환하고, 스프레이 로딩 전략도
아래에서 함께 확정한다(결정 5·6번).

## 결정

1. **저작권**: Valve의 Dust2 원본 지오메트리·텍스처·머티리얼을 참조하지
   않고 원본 제작한다. "Dust2 분위기"는 **색조·재질의 유사**(사암색
   팔레트, 건조한 지중해풍 재질)로 한정하며, **레이아웃은 독자 설계**다
   (RQ-30 확정). 원본 게임의 참고 이미지·영상을 자산 소스로 직접
   트레이싱·리토폴로지하지 않는다.
2. **저작 포맷**: Blender에서 모델링 → **glTF(.glb, 바이너리 단일 파일)**로
   익스포트. glTF는 Three.js/R3F의 1급 임포트 포맷이라 클라이언트 렌더링
   파이프라인과 직결된다.
3. **충돌 메시 분리**: 시각 메시(고폴리, 텍스처·머티리얼 포함)와 물리
   충돌 메시(저폴리, 박스/캡슐/컨벡스헐 근사)를 별도 지오메트리로 분리
   제작한다. 서버([ADR-0004](0004-physics-collision.md), Rapier)는 충돌 메시만 로드해
   물리 월드를 구성하고, 클라이언트는 시각 메시를 렌더링하되 예측용
   충돌 메시도 동일하게 로드한다(서버-클라이언트 동일 충돌 데이터,
   [ADR-0004](0004-physics-collision.md)의 결정론 요구와 정합).
4. **특수 볼륨의 명명 규칙**: 스폰 지점(RQ-31 Safe Zone 겸용), 사다리
   (RQ-21), 박스 점프 발판(RQ-22 연계)은 glTF 노드의 명명 규약(예:
   `Spawn_*`, `Ladder_*`)으로 마킹된 빈 노드(empty)/저폴리 볼륨으로
   익스포트하고, 서버가 맵 로드 시 이름 규칙을 파싱해 게임플레이 볼륨
   (트리거/센서 콜라이더)으로 등록한다. 씬 그래프 안에 게임플레이
   메타데이터를 자연 배치해 별도 레벨 에디터 없이 Blender만으로 배치·
   조정 가능하게 한다.
5. **맵 규모·배치(RQ-30~32 확정값)**:

   | 항목 | 값 |
   |---|---|
   | 플레이 면적 | 약 60×60m |
   | 스폰 지점 수 | 14~16개, 직전 사용 지점을 회피하는 순환 로테이션 |
   | Safe Zone 반경 | 스폰 지점 반경 5m, 이탈 즉시 보호 해제 |
   | Safe Zone 사격 제한 | 내부에서는 사격 불가(무기 비활성화) |
   | 사다리 | 2개 |
   | 박스 클러스터 | 3곳, 클러스터당 4~6개 박스 |

   박스 높이는 점프 높이 1.0m([ADR-0004](0004-physics-collision.md), RQ-92)로 등반
   가능한 치수여야 한다(RQ-32) — 박스의 정확한 치수는 req가 정하지 않아
   맵 에셋 자체의 콜라이더 치수(설정값)로 둔다(값 발명 금지). Safe Zone의
   "내부 사격 불가"는 물리 볼륨이 아니라 무기 시스템의 관심사이므로 이
   ADR은 볼륨 마킹(위 4번)만 제공하고, 사격 비활성화 로직 자체는
   [ADR-0005](0005-hit-registration.md)/전투 로직 구현이 담당한다.
6. **스프레이 비트맵 100종 로딩 전략(RQ-42/RQ-94 확정): 텍스처 아틀라스
   채택.** 100종을 사전 합성한 단일(또는 소수의) 아틀라스 시트로
   프리로드한다. RQ-94가 스프레이를 **AI 생성 고정 세트**(사용자 업로드
   없음)로 확정해, 원본 해상도·포맷을 아트 파이프라인이 아틀라스 셀
   크기에 맞춰 사전 조정할 수 있게 됐다 — 이전에 미결이었던 "알 수 없는
   원본 소스" 문제가 해소됐다. 아틀라스 셀 해상도·시트 분할 수는
   구현 시 아트 파이프라인 산출물에 맞춘 설정값으로 정한다(값 발명 금지).

## 근거

- glTF는 Three.js/R3F 생태계의 사실상 표준 포맷 — 로더 신뢰성·PBR
  머티리얼 지원·바이너리 단일 파일(.glb)로 배포 자산 관리가 단순하다.
- 충돌/시각 메시 분리: 시각 메시(수만 폴리곤)를 그대로 물리 충돌에 쓰면
  서버 Rapier의 콜라이더 생성·레이캐스트 비용이 커지고(30Hz 틱 예산
  안에서 수행해야 함, RQ-60), 정밀한 시각 디테일(장식, 얇은 구조물)이
  원치 않는 충돌(끼임, 미세 턱)을 유발한다 — 저폴리 근사가 표준 관행이다.
- 명명 규약 기반 볼륨: 씬 그래프 자체를 게임플레이 데이터 저장소로 쓰면
  Blender 뷰포트에서 스폰/사다리 배치를 반복 조정할 수 있다 — 별도 데이터
  포맷(좌표 수기 입력)보다 시각적 배치·수정이 빠르다.
- **스프레이 로딩 전략 확정 근거(RQ-42/RQ-94)**: 100개 개별 텍스처를 각각
  별도 머티리얼로 로드하면 스프레이 사용마다 드로우콜이 분리되고(배칭
  불가), 첫 화면에서 100개 텍스처 요청이 초기 로딩 비용을 늘린다.
  텍스처 아틀라스로 묶으면 드로우콜과 초기 요청 수를 하나로 줄인다 —
  스프레이는 표면 위 작은 데칼로 렌더링되어 근접 정밀 디테일이 요구되지
  않으므로(탄흔·혈흔과 동일한 사용 맥락, RQ-70/71), 아틀라스 분할로 인한
  셀당 해상도 손실이 체감 품질에 미치는 영향이 제한적이다.
- **AI 생성 에셋의 라이선스(RQ-94)**: 스프레이 100종을 AI 생성 모델로
  제작하므로, 사용한 생성 모델의 이용 약관이 산출물의 재배포·(잠재적)
  상업적 이용을 허용하는지 아트 파이프라인 착수 전에 확인해야 한다 —
  이 ADR은 그 확인을 전제 조건으로 명문화한다. 어떤 생성 모델을 쓸지,
  그 라이선스 조항의 구체 내용은 이 문서가 정하지 않는다(모델 선택은
  구현 단계 사안) — 다만 재배포·상업적 이용을 금지하는 모델의 산출물은
  채택할 수 없다는 제약만 여기 남긴다.
- 버린 대안:
  - **참고 이미지/영상을 3D 스캔·포토그래메트리 소스로 사용**: 원본을
    직접 캡처하지 않더라도, 팬 리메이크 자료(원본 치수·텍스처를 그대로
    딴 모델)를 참조하면 저작권 회피(RQ-30)가 실질적으로 무력화된다 —
    채택 불가.
  - **단일 메시(시각=충돌 겸용)**: 구현은 단순하지만 위 근거의 성능·품질
    문제로 배제.
  - **별도 레벨 에디터/데이터 포맷(JSON으로 볼륨 좌표 수기 관리)**:
    Blender 뷰포트 배치보다 반복 조정 비용이 크다(좌표를 눈대중으로
    입력·미세조정하려면 익스포트-확인 왕복이 늘어난다). 맵이 다수화되면
    전용 에디터가 필요해질 수 있으나, v1(맵 1종, RQ-30)에서는 과잉이다.
  - **스프레이 개별 텍스처 프리로드(비아틀라스)**: 위 트레이드오프에서
    설명한 드로우콜 분리·초기 로딩 비용 증가로 채택하지 않는다.
  - **스프레이 지연 로드(최초 사용 시 요청)**: 스프레이는 F 키 입력 시
    즉시 표면에 부착돼야 하는 저지연 상호작용(RQ-42)이라, 네트워크 요청
    왕복을 인터랙션 경로에 끼워 넣으면 체감 지연이 생긴다 — 프리로드가
    이 즉시성 요구에 더 맞는다.

## 결과

- "Dust2 분위기"의 해석이 결국 제작자의 판단에 달려 있다 — 저작권 회피가
  "치수·텍스처를 베끼지 않았다"는 사실만으로 완전히 보장되지 않고, 최종
  상표/트레이드드레스 유사성 판단은 법적 소양이 필요한 영역이다. 이 ADR은
  기술적 파이프라인(원본 미참조 제작)만 규정하며, 법적 리스크 평가 자체는
  범위 밖이다 — 후회 지점: 완성 후 특정 랜드마크(다리·터널 배치 등)가
  원본과 지나치게 흡사하다는 지적이 나올 수 있다.
- 명명 규약(`Spawn_*`, `Ladder_*` 등)이 확정되면 이후 모든 맵 자산이 이
  규약을 따라야 한다 — 규약 변경은 기존 .glb 자산 전체의 재작업을
  수반한다.
- 맵이 1종(RQ-30)이라 파이프라인은 "여러 맵 관리"를 고려하지 않았다 —
  맵 추가가 필요해지면 로딩/선택 UI, 맵별 설정 분리가 추가로 필요하다.
- 🟡 확정 필요: 정확한 명명 규약 문자열과 박스 점프(RQ-22)에 필요한 별도
  볼륨 타입 유무는 구현 착수 시 확정한다(req가 정하지 않은 세부이므로 이
  ADR 승인과 별개로 구현 시 함께 명문화).
- 스프레이 아틀라스가 확정되어 `harness/workflow/fe.md`의 에셋 로딩 절과
  이 ADR이 더 이상 미결로 남지 않는다 — 아틀라스 생성은 빌드 타임
  산출물(에셋 파이프라인 스크립트)로 다뤄야 하고, 아직 그 스크립트는
  작성되지 않았다(구현 단계 산출물). AI 생성 모델의 라이선스 확인이
  끝나지 않으면 실제 스프레이 자산 제작에 착수할 수 없다 — 아트
  파이프라인 착수 전 확인 항목으로 남는다.
- **잔존 미해결 사항(승인 상태와 무관)**: 실제 Blender→glTF 익스포트
  파이프라인과 이름 파싱 서버 코드, 스프레이 아틀라스 생성 스크립트가
  아직 검증되지 않았다 — RQ-30~32/RQ-42/RQ-94 확정으로 이 ADR을
  승인하지만, 파이프라인 자체의 구현 검증은 구현 단계 작업으로 남는다.
