# ADR-0008: 테스트 전략 — TDD, Vitest, 결정론적 헤드리스 시뮬레이션

- 상태: 승인
- 날짜: 2026-07-21
- 관련 스펙: 전 RQ (특히 RQ-60~64 넷코드, RQ-12~18 전투 로직)

## 맥락

TDD(Red→Green→Refactor)가 이 하네스의 규칙이다. 채팅 앱(bvwebchat)과
달리 이 프로젝트는 실시간 3D 렌더링(R3F)과 물리 시뮬레이션(Rapier)이
핵심 도메인 로직과 얽혀 있다 — 렌더링은 결정적으로 단위 테스트하기
어렵고, 반대로 데미지·틱 로직은 순수 함수로 격리 가능해야 한다. 이 ADR은
무엇을 어떤 레벨로 테스트하고 무엇을 범위 밖에 둘지 결정한다.

## 결정

1. **레벨**:
   - **단위 테스트**: 데미지/탄도 계산(헤드샷 배율 RQ-13, 낙하 데미지
     RQ-18, HP 감산·사망 판정 RQ-14), 랙보상 되감기 인덱스 계산(RQ-64),
     스폰 보호 타이머(RQ-16), AFK 타이머(RQ-43) 등 순수 함수/상태 전이
     로직.
   - **통합 테스트**: 넷코드·룸 동작을 **Colyseus 룸 경계**에서
     검증한다 — 테스트 프로세스 내에 Colyseus 서버를 기동하고 클라이언트
     SDK(colyseus.js)로 접속해 실제 join/leave, 상태 동기화(Schema),
     메시지(채팅 RQ-40, 사격 RQ-12) 왕복을 확인한다.
   - **스모크**: 배포된 이미지([ADR-0009](0009-deployment.md)) 대상 헬스체크 + 핵심
     골든(룸 접속, 채팅 전파) 재실행.
2. **결정론 요구(게임 특화 핵심)**: 시뮬레이션 틱 함수는 **순수 함수**로
   작성한다 — `tick(state, inputs, dt) => newState` 형태로, 실제
   시간(`Date.now()`, `performance.now()`)이나 실제 타이머(`setInterval`)에
   직접 의존하지 않고 시간은 인자로 주입받는다. 테스트는 Vitest의 fake
   timer 또는 수동 시간 주입으로 N틱을 결정적으로 진행시켜 결과를
   단언한다. 이 요구가 없으면 넷코드 테스트가 실 타이머에 의존해
   flaky해지고, 사격 판정처럼 타이밍이 결과를 좌우하는 로직을 재현
   가능하게 검증할 수 없다.
3. **러너**: **Vitest** — Vite 클라이언트와 생태계 일관, TS 네이티브,
   fake timer 내장. 테스트 위치: `tests/unit/`(순수 로직),
   `tests/integration/`(Colyseus 룸 경계). 테스트 이름에 RQ-ID 포함.
4. **Red 증거**: 테스트 커밋이 구현 커밋보다 선행하고 PR에 Red 실행
   출력(`vitest run` + `tsc --noEmit`)을 첨부한다. tsc 오류가 아직 만들지
   않은 src/ 모듈 임포트(TS2307/TS2305)에 국한되면 정당한 Red다(그린필드
   신규 모듈은 필연적으로 이 오류를 낸다).
5. **테스트 더블**:
   - 넷코드 통합 테스트는 Colyseus 서버를 in-process로 기동한다(실
     네트워크 스택은 불요하나 localhost WebSocket은 허용 — 클라이언트
     SDK가 실제 WebSocket 연결을 요구하기 때문에 완전한 목은 어렵다.
     포트는 임의(0) 바인딩, 모든 대기에 타임아웃 상한을 명시해 flaky를
     억제한다). 그럼에도 실 WebSocket 통합은 vitest/Node 워커 teardown
     계층에서 ~2% 콜드스타트 워커 크래시(테스트 로직 아님, 근본 원인
     black-box)가 남는다 — `scripts/check.sh`가 **통합 테스트만** 크래시
     시그니처에 게이팅해 최대 3회 재시도한다. 단언 실패는 재시도 대상이
     아니라 즉시 하드 실패다(검증은 약화하지 않는다). 경위: RQ-04,
     `harness/progress.md` 19a.
   - 물리(Rapier)는 실제 라이브러리를 그대로 사용한다(대역 금지) — 물리
     엔진 자체를 목으로 대체하면 충돌·낙하 데미지 로직의 검증력이
     사라진다. 대신 결정론 요구(2번)로 fake timer와 조합해 재현성을
     확보한다.
   - Redis/SQLite([ADR-0006](0006-identity-and-persistence.md))는 통합 테스트에서
     최소한 SQLite를 `:memory:` 모드로 대체한다. Redis 대체 방식(in-memory
     mock vs 실제 Redis 컨테이너)은 구현 단계에서 확정한다.
6. **렌더링(R3F)은 단위/통합 테스트 대상이 아니다** — WebGL 렌더링
   결과(픽셀)·Three.js 씬 그래프의 시각적 정확성은 자동화된 단위 테스트로
   검증하지 않는다. 검증은 스모크(기동 확인)와 수동 확인(개발자 육안
   검토)으로 한다. 근거: 헤드리스 WebGL 렌더링 검증은 비용 대비 신뢰도가
   낮고(스냅샷 테스트는 GPU/드라이버 차이로 flaky), 이 프로젝트의 핵심
   리스크는 렌더링 결함이 아니라 넷코드·전투 로직의 결함이다(RQ-61 서버
   권위가 정확성의 실제 진실 공급원). 이 경계는 `harness/workflow/fe.md`
   (FE 구현 워크플로우)가 이미 전제하고 있는 검증 표(타입체크·lint·
   스모크·스크린샷 수동 확인)와 동일하다 — 게이트 세부는 fe.md가 정본이며
   여기서 반복하지 않는다.
7. **속도 예산**: `check.sh --fast` <5초(변경 파일 lint만), CI 게이트
   <3분(전체 lint+typecheck+test). 초과분(예: 다틱 장시간 시뮬레이션
   안정성 테스트)은 게이트 밖 수동/야간 실행으로 넘긴다.
8. **예외**: 문서·설정·순수 스타일 변경, 맵 에셋(.glb) 추가는 TDD 면제.
   스파이크(탐색)는 `spike/*` 브랜치에서만 — 머지 금지, 교훈만 가져와
   TDD로 재구현.
9. **커버리지**: 수치 게이트 없음 — 테스트 선행률(test-first rate)만
   추적한다. 커버리지는 게이트가 되는 순간 게임의 대상이 된다.

## 근거

- 이 프로젝트의 검증력은 두 경계에 있다: (1) 순수 로직(데미지/틱 계산)의
  단위 정확성, (2) Colyseus 룸 경계에서의 통합 동작(입장/퇴장/동기화/
  메시지). 렌더링은 검증력의 원천이 아니다 — 프레임이 잘못 그려져도
  게임 로직은 정확할 수 있고, 반대(로직 결함)가 이 프로젝트에서 훨씬
  치명적이다(불공정 판정·치팅으로 직결).
- 결정론(순수 tick 함수 + 주입 시간)은 게임 넷코드 테스트의 표준 요구다 —
  실 타이머·실 RAF에 의존하면 30Hz 틱 경계의 미세한 시간차가 테스트
  결과를 좌우해 flaky해진다. bvwebchat도 fake timer를 썼지만(ADR-0005),
  이 프로젝트는 한 걸음 더 나가 "틱 함수 자체가 순수해야 한다"는 아키텍처
  제약으로 명문화한다 — 그래야 실제 서버 루프 코드와 테스트 대상 로직이
  같은 함수가 된다.
- 버린 대안:
  - **렌더링까지 단위 테스트(예: react-three-test-renderer + 스냅샷)**:
    WebGL 컨텍스트 목킹이 무겁고, 이 규모 프로젝트에서는 스냅샷 diff보다
    육안 검토가 신뢰도 대비 비용이 낮다 — 과잉 투자로 판단해 배제.
  - **E2E(Playwright) 상시 실행**: 브라우저 기동+3D 렌더링+WebSocket
    연결까지 묶으면 매우 느리고 flaky — bvwebchat과 동일 이유로 배제,
    필요 시 스모크로만 한정.
  - **틱 함수가 내부적으로 `Date.now()`를 직접 호출(구현 편의)**: 당장은
    단순해 보이지만 테스트가 fake timer(vi.useFakeTimers + 실제 타이머
    트리거)에 의존하게 되어 "몇 틱을 얼마나 진행시켰는지"의 재현성이
    Vitest 타이머 목 구현 세부에 종속된다. 순수 함수 + 시간 주입이 더
    직접적이고 러너 의존도가 낮다.

## 결과

- 신규 게임 로직(데미지, 이동, 랙보상)을 작성할 때 항상 "이 함수가
  실시간·타이머·네트워크에 직접 의존하는가"를 먼저 점검해야 한다 —
  의존하면 순수 함수로 리팩터링 후 얇은 어댑터(실제 setInterval 호출
  등)로 감싸는 계층 분리가 강제된다. 초기 구현 속도는 다소 느려지지만
  (추상 계층 하나 추가), 넷코드 전체가 결정적으로 재현 가능해진다.
- 렌더링 결함(예: 특정 각도에서 총구 이펙트가 잘못 나옴)은 자동 테스트로
  잡히지 않는다 — 수동 QA·스모크 확인에 의존하며, 이는 회귀를 놓칠 수
  있는 명시적으로 수용된 공백이다. FE 쪽 게이트 세부(타입체크/lint/스모크/
  스크린샷 수동 확인 표)는 `harness/workflow/fe.md`를 참고한다 — 이
  ADR은 그 경계 결정과 근거만 소유하고 표는 중복 기재하지 않는다.
- Colyseus 룸 통합 테스트가 실제 WebSocket(localhost)에 의존하므로 완전한
  격리는 아니다 — 포트 충돌·타임아웃 관리가 필요하다(bvwebchat도 동일
  제약을 안고 있었다).
- 후회 지점: 물리(Rapier)를 실제로 돌리는 통합 테스트는 CPU 비용이 순수
  로직 테스트보다 크다 — 테스트 스위트가 커지면(맵·엔티티 수 증가) 3분
  CI 게이트를 압박할 수 있고, 그때는 물리 테스트를 별도 느린 스위트로
  분리해야 할 수 있다.
