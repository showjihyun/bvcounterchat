# ADR-0003: 넷코드 권위 모델 — 30Hz 고정 틱, 서버 권위, 클라이언트 예측+보간

- 상태: 승인 (근거: docs/req/06)
- 날짜: 2026-07-21
- 관련 스펙: RQ-60, RQ-61, RQ-62, RQ-63, RQ-92

## 맥락

req/06이 30Hz 고정 틱, 서버 권위, 클라이언트 예측, 보간, 랙보상을 이미
못박았다(RQ-60~64). 이 ADR은 그중 틱·권위·예측·보간의 배선 방식을
구체화한다(랙보상 자체는 [ADR-0005](0005-hit-registration.md)에서 다룬다).
Deep Interview(질문 8)로 RQ-92가 **공중 가속 미허용**(에어 스트레이프·
버니합 없음)을 확정했다 — 이 결정이 예측·재조정의 결정론적 정확도에
주는 영향을 아래 결과에 반영한다.

## 결정

서버는 **30Hz(≈33.3ms) 고정 델타**로 입력·물리 시뮬레이션을 진행하고 매 틱
상태 스냅샷을 브로드캐스트한다(RQ-60/61). 클라이언트는 자신의 입력을
로컬에 즉시 반영하는 **클라이언트 예측**을 수행하고, 서버 스냅샷 도착 시
서버 상태와의 차이를 **화해(reconciliation)**한다(RQ-62). 다른 플레이어의
원격 엔티티는 서버 스냅샷 사이를 **보간(entity interpolation)**해
표시한다(RQ-63).

## 근거

- **입력 커맨드 버퍼**: 클라이언트는 각 입력에 시퀀스 번호를 붙여 서버로
  전송하고, 서버는 틱 순서대로 소비해 상태를 갱신한 뒤 처리된 마지막
  시퀀스 번호를 스냅샷에 담아 반환한다. 클라이언트는 그 번호 이전의 로컬
  예측을 서버 상태로 교체하고, 이후 미확인 입력을 재생(replay)해 예측을
  재구성한다 — FPS 넷코드의 표준 패턴(Quake/Source 계열과 동일 계열).
- **스냅샷 레이트 vs 틱 레이트**: 스냅샷은 매 틱(30Hz)마다 생성한다. v1은
  단순성을 위해 전송 레이트를 틱 레이트와 동일(30Hz)하게 시작하고, 대역폭
  문제가 실측되면 전송만 낮은 레이트로 분리한다.
- **보간 지연 버퍼**: 원격 엔티티는 항상 최신 스냅샷보다 한 스냅샷 간격
  이상(약 33~66ms) 뒤쳐진 시점을 그린다 — 두 스냅샷 사이를 보간할 여유를
  확보해야 스터터 없이 매끄러운 움직임이 나온다. 이 지연은 "내가 보는
  상대는 이미 과거"라는 뜻이며, 사격 랙보상([ADR-0005](0005-hit-registration.md))과
  개념적으로 짝을 이룬다 — 내가 조준한 순간의 대상 위치로 서버가 되감아
  판정한다.
- **공중 가속 미허용(RQ-92)이 넷코드 설계에 주는 영향**: 공중에서 입력에
  따라 궤적이 분기하지 않으므로(점프 시점 수평 관성만 유지), 예측 재생이
  다뤄야 할 입력-상태 결합이 단순해진다 — 자세한 영향은 결과 절 참고.
- 버린 대안:
  - **가변 틱(variable timestep)**: 물리 결정성이 깨지고 예측 재생의
    재현성이 사라진다. req/06이 고정 틱(30Hz)을 이미 명시.
  - **클라이언트 권위(클라이언트 보고 상태 신뢰)**: RQ-61이 명시적으로
    금지한다("클라이언트가 보고한 상태는 그대로 반영하지 않아야 한다").
  - **외삽(dead reckoning)만 사용, 보간 생략**: 방향 전환 시 오버슈트가
    크고 정정(snap correction)이 눈에 띈다. RQ-63이 보간을 명시적으로
    요구.

## 결과

- 보간 지연 버퍼만큼 원격 플레이어는 항상 "약간 과거"로 보인다 — 저지연
  환경에서도 0은 아니다. [ADR-0005](0005-hit-registration.md)의 되감기 상한과 일관되게
  튜닝해야 한다.
- 클라이언트 예측(이동)과 서버 시뮬레이션이 같은 규칙(중력·충돌·속도)을
  두 곳에 구현해야 한다 — 규칙이 갈라지면 예측 오차가 커지고 정정이 자주
  보여 "미끌거림"으로 체감된다. [ADR-0004](0004-physics-collision.md)이 이 정합
  문제(서버·클라이언트 동일 물리 엔진)를 다룬다.
- 입력 커맨드 버퍼·시퀀스 번호 프로토콜은 Colyseus의 기본 메시징 위에
  프로젝트가 직접 얹는 애플리케이션 레벨 레이어다 — [ADR-0002](0002-realtime-transport.md)의
  스코프를 벗어나는 설계·구현이 별도로 필요하다.
- **RQ-92 확정(공중 가속 미허용)이 예측 정확도에 주는 영향**: 공중 궤적이
  오직 "점프 이함 시점의 수평 속도 + 경과 시간 + 중력"만으로 결정되고
  입력에 반응하지 않으므로, 클라이언트와 서버가 같은 궤적을 재현하는 데
  필요한 상태가 최소화된다 — 예측 재생이 입력 시퀀스의 미세한 타이밍
  차이나 프레임레이트 편차에 영향받지 않고 결정론적으로 수렴한다. 만약
  공중 가속(에어 스트레이프)을 허용했다면 공중에서도 매 틱 입력을
  반영해야 해 클라이언트-서버 부동소수 연산 순서 차이가 누적되고, 예측
  오차·정정 스냅이 늘었을 것이다. CS류 이동 숙련도(공중 조작)라는 게임성
  축을 포기한 대가로 넷코드 신뢰성을 얻은 명시적 트레이드오프다(RQ-92
  근거, 질문 8). 이동 파라미터의 구체 수치(6 m/s 기본 속도 등)는
  [ADR-0004](0004-physics-collision.md)의 캐릭터 컨트롤러 설정값으로 반영된다 — 이
  ADR은 예측·재조정 메커니즘만 다룬다.
