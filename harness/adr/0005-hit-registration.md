# ADR-0005: 히트 판정 — 서버 hitscan + 랙보상(되감기 상한 200ms)

- 상태: 승인 (근거: RQ-90, RQ-64 확정 — 2026-07-21 Deep Interview)
- 날짜: 2026-07-21
- 관련 스펙: RQ-12, RQ-13, RQ-55, RQ-64, RQ-90

## 맥락

RQ-12가 서버 hitscan 판정을, RQ-13이 헤드샷 2배를, RQ-64가 사수 RTT만큼
대상을 되감는 랙보상을 요구한다. req가 애초 침묵했던 부분: 히트박스 모델의
세부 구성, 되감기 버퍼 길이·상한 값, 무기 파라미터, 그리고 "엄폐 뒤에서
맞았다"류 공정성 트레이드오프를 어떻게 다룰지. Deep Interview(질문
1~4·28·29)로 RQ-90(무기 파라미터)과 RQ-64(되감기 상한 200ms, RTT 150ms
보장)이 확정되어 이 ADR을 승인으로 전환한다.

## 결정

- 사격은 클라이언트가 발사 이벤트(조준 방향, 타임스탬프/시퀀스)만 서버에
  보내고, 서버가 그 정보를 근거로 판정 레이(ray)를 쏴 hitscan 판정을
  수행한다(RQ-12).
- 히트박스는 **2단 구성**: 헤드(작은 볼륨) + 바디(나머지 전신) — 헤드
  볼륨 명중 시 RQ-13의 2배 데미지를 적용한다. 팔다리 별도 배율은 두지
  않는다(req가 헤드/기본만 구분).
- **무기 파라미터(RQ-90 확정)**: 바디 데미지 25(4타 킬), 헤드 데미지
  50(2타 킬, RQ-13의 2배 규칙에서 유도), 연사 간격 150ms(400 RPM),
  사거리 감쇠 없음(전 사거리 고정 데미지). 서버는 같은 플레이어의 발사
  요청 간격이 150ms 미만이면 무시한다(rate-limit) — 클라이언트가 연사
  속도를 조작해 서버에 보고하는 경로를 차단한다.
- **탄퍼짐(RQ-90 확정)**: 랜덤 콘 패턴 — 콘 반경 내 균등분포로 착탄점을
  흩뿌린다. 난수는 **서버가 발급한 시드**로 생성해야 하며, 시뮬레이션
  코드는 `Math.random()`을 직접 호출하지 않는다([ADR-0008](0008-test-strategy.md)의
  결정론 요구와 일치). 콘 반경의 구체값과 이동·점프 시 정확도 저하
  계수는 req가 수치를 정하지 않았다 — 코드가 아닌 설정 파일의 조정
  가능한 값으로 둔다(값 발명 금지).
- **랙보상(RQ-64 확정)**: 서버는 링버퍼에 최근 6틱(30Hz 기준 200ms 분)의
  각 플레이어 히트박스 위치 스냅샷을 보관한다. 사격 판정 시 사수의
  RTT(또는 사수가 마지막으로 확인한 서버 틱)만큼 과거로 대상들의
  히트박스를 되감아 그 시점 기준으로 레이캐스트한다. 되감기 상한은
  **200ms**로 확정하며, 이를 초과하는 RTT의 사수는 상한에서 절단된
  위치로 판정된다.

## 근거

- 서버 판정 + 랙보상은 "조준했을 때 맞았다"는 사수 체감과 "서버가 유일한
  진실"이라는 RQ-61 원칙을 동시에 만족하는 절충안이다(Source 엔진류
  랙보상 문서화 패턴과 동일 계열) — 클라이언트 판정 신뢰(RQ-12가 이미
  금지)나 랙보상 없는 순수 현재-상태 판정보다 사수 체감이 낫다.
- **되감기 상한 200ms 확정 근거**: RTT 150ms 보장(RQ-64)에 여유(50ms)를
  더한 값이다 — 정상적인 네트워크 조건(사내망, RQ-80)의 사수는 상한에
  걸리지 않으면서도, 링버퍼 크기(6틱 분의 히트박스 히스토리)가 과도하게
  커지지 않는 균형점이다(질문 28·29 근거).
- **서버 시드 기반 탄퍼짐 근거**: 클라이언트가 자체 난수로 탄퍼짐을
  계산해 서버에 착탄 좌표만 보고하면 클라이언트가 결과를 조작할 수
  있다(RQ-61 위반). 서버가 시드를 발급하고 서버 쪽에서 콘 내 착탄점을
  계산해야 서버 권위가 유지된다.
- 되감기 상한을 두는 이유: 상한이 없으면 RTT가 큰(또는 타임스탬프가
  조작된) 클라이언트가 임의로 먼 과거를 요구해 "이미 엄폐한 상대가
  뒤늦게 맞는" 체감을 과도하게 키우고, 서버가 보관할 히스토리 버퍼도
  무한정 커진다 — 링버퍼 크기(=되감기 상한)를 고정해 메모리와 공정성을
  함께 제한한다.
- 버린 대안:
  - **랙보상 없음(서버 "현재" 상태로만 판정)**: 저지연에서도 사수가 조준한
    순간과 서버 판정 순간 사이에 RTT만큼 대상이 이동해 있어 "분명히
    맞혔는데 안 맞았다"는 체감이 커진다. req/06이 랙보상(RQ-64)을 이미
    요구하므로 채택 불가.
  - **클라이언트 판정 신뢰(클라이언트가 명중을 주장, 서버는 확인만)**:
    RQ-12가 명시적으로 금지 — 서버 권위 원칙 위반, 치트 시 즉시 악용된다.
  - **되감기 상한 없음(RTT 그대로 무제한 반영)**: 위의 엄폐 공정성 문제와
    버퍼 무한 확장 문제로 채택 불가.
  - **헤드/바디 이상의 다단 히트박스(팔·다리 별도 배율)**: req가 헤드샷
    배율만 요구 — 그 이상은 과잉 설계(YAGNI)로 배제.
  - **고정 스프레이 패턴(비랜덤, 학습 가능한 반동 궤적)**: Counter-Strike류의
    학습형 고정 패턴은 정교한 반동 곡선 설계가 필요하다 — 인터뷰
    (질문 4)가 랜덤 콘을 명시적으로 선택했다.

## 결과

- **"엄폐 뒤에서 맞는" 공정성 트레이드오프**: 랙보상이 사수에게 유리하게
  작동하므로, 대상 플레이어 입장에서는 자신의 화면에서 이미 엄폐물 뒤로
  이동한 후에도 서버가 과거 시점(엄폐 전 위치)으로 되감아 판정하면 피격
  판정을 받을 수 있다 — 랙보상의 잘 알려진 부작용이며, 이 프로젝트가
  의도적으로 수용하는 트레이드오프다. 200ms 상한이 이 체감의 최대 폭을
  제한한다.
- 링버퍼(되감기 버퍼)는 플레이어당 6틱 분의 히트박스 트랜스폼을 보관하는
  메모리 비용을 발생시킨다 — 10인 규모에서는 무시할 수준이다.
- **킬 이벤트 스키마의 headshot 플래그**: RQ-55(킬 피드가 헤드샷 킬을
  시각적으로 구분해 표시해야 함, 요구사항 v1.0에서 이 ADR을 직접 지목)를
  만족하려면 클라이언트가 스스로 헤드샷 여부를 재도출할 수 없다 — 명중
  부위는 서버의 권위적 레이캐스트(RQ-12/13)에서만 확정되고, 클라이언트는
  그 판정 과정(레이·히트박스 교차 계산)을 보지 못한다. 따라서 서버가
  브로드캐스트하는 킬 이벤트([ADR-0002](0002-realtime-transport.md)의 Colyseus
  Schema/메시지)는 가해자·피해자 식별자 외에 `headshot: boolean` 필드를
  명시적으로 포함해야 한다 — 이 필드 없이는 RQ-55의 클라이언트 구현
  자체가 불가능하다.
- 서버의 150ms rate-limit(연사 속도 강제)은 무기가 Pistol 1종뿐인
  v1(비요구 목록)에서는 단순하지만, 향후 무기가 추가되면 무기별 연사
  간격 테이블로 확장해야 한다.
- **잔존 미해결 사항(승인 상태와 무관)**: 히트박스 세부 치수(캡슐 반지름/
  높이, 헤드 볼륨 크기)는 캐릭터 모델 에셋이 아직 없어 확정할 수 없다 —
  RQ-90/RQ-64 수치 확정으로 이 ADR을 승인하지만, 정확한 치수는 캐릭터
  모델 제작 후 코드가 아닌 설정 파일 값으로 확정한다(값 발명 금지).
