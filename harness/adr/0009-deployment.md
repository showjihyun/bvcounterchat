# ADR-0009: 배포 — Docker(서버 이미지) + Nginx(HTTP/WS) + Redis, 멀티 컨테이너 단일 호스트

- 상태: 승인 (근거: docs/req/03, RQ-80 확정)
- 날짜: 2026-07-21
- 관련 스펙: RQ-80, RQ-81, RQ-04

## 맥락

req/03·RQ-80이 Docker+Nginx 배포를 이미 지정했다. RQ-81(Redis+SQLite)이
추가로 스테이트풀 의존성을 요구해([ADR-0006](0006-identity-and-persistence.md)),
"단일 이미지"가 "단일 컨테이너 프로세스"를 뜻하는지 "단일 호스트의 여러
컨테이너"를 뜻하는지 명확히 해야 한다. Deep Interview(질문 27·30~32)로
RQ-80이 배포 대상을 **사내망/로컬 전용**으로, 서비스 프로토콜을 **HTTP/WS
(TLS 불요)**로, 무중단 배포를 **비목표**로 확정했다. RQ-04(상시 접속
단일 상설 세션)는 배포가 장기 실행을 전제해야 함을 암시한다.

## 결정

1. **컴포지션**: 게임 서버(Node.js + Fastify + Colyseus,
   [ADR-0002](0002-realtime-transport.md))를 하나의 Docker 이미지로 빌드한다. Redis·
   SQLite는 별도 관심사로 분리한다: Redis는 별도 컨테이너(공식 이미지)로,
   SQLite는 파일 기반이므로 게임 서버 컨테이너의 볼륨 마운트 경로에
   위치한다. 즉 "단일 이미지"는 애플리케이션 서버 이미지를 뜻하며, 전체
   배포 단위는 docker-compose(또는 동등한 멀티 컨테이너 구성)로 Nginx+
   서버+Redis 3개 컨테이너를 한 호스트에 구성한다.
2. **Nginx — HTTP/WS 확정(RQ-80)**: 리버스 프록시로 WebSocket 업그레이드
   (Colyseus의 `Upgrade: websocket` 헤더)를 게임 서버 컨테이너로
   프록시하고, 정적 클라이언트 자산(빌드된 R3F 앱)을 직접 서빙한다.
   **사내망/로컬 전용 배포(RQ-80 확정)이므로 평문 HTTP/WS로 서비스하며
   TLS를 요구하지 않는다.** 다만 Nginx 설정은 서버 블록을 분리하고
   인증서 경로를 환경변수/볼륨으로 빼둬, 나중에 `listen 443 ssl`과
   인증서만 추가하면 TLS 종단을 붙일 수 있도록 구성을 열어둔다(전환
   비용을 낮추는 목적이며, 지금 인증서를 준비하거나 발급받지는 않는다).
3. **정적 자산 서빙 위치**: Nginx가 `dist/client`를 직접 서빙한다(게임
   서버 프로세스에 정적 서빙 책임을 얹지 않음) — Nginx가 이미 리버스
   프록시로 앞단에 있으므로 정적 파일 서빙은 Nginx의 표준 역할에 맡기고,
   Node 프로세스는 Colyseus 룸 로직 + Fastify REST(헬스체크 등)에
   집중한다.
4. **상태 지속성**: RQ-81에 따라 Redis(세션)는 재시작 시 소실을 허용 —
   볼륨 마운트 없이 컨테이너 재시작 시 초기화되어도 무방하다. SQLite(통계)
   는 named volume에 파일을 두어 게임 서버 컨테이너 재시작·재배포에도
   통계 파일이 보존되게 한다.
5. **무중단 재배포 — 비목표로 확정(RQ-80)**: 재배포 시 접속 중 플레이어는
   끊긴다는 것을 명시적으로 수용한다 — RQ-80이 "무중단 배포는 목표가
   아니다"라고 확정했으므로, 이는 더 이상 제안 단계의 트레이드오프가
   아니라 확정된 요구사항이다. 재연결 유예([ADR-0006](0006-identity-and-persistence.md))가
   짧은 재시작(수 초)을 흡수할 수 있으나, 보장하지 않는다.

## 근거

- 단일 이미지(게임 서버) + 별도 Redis 컨테이너 구성이 RQ-80 "Docker
  이미지로 패키징되어 Nginx 뒤의 사내망 단일 호스트에 배포 가능"을
  만족한다 — "단일 호스트"는 충족하되 "단일 컨테이너"까지 강제하지는
  않는다(req 문구가 호스트 단위이지 컨테이너 단위가 아니다). Redis를
  게임 서버 프로세스 내부에 폴리필하면 req/03이 명시한 기술 선택 자체를
  어기게 된다.
- Nginx가 정적 자산을 서빙: 게임 서버(Node)가 정적 서빙까지 겸하면
  코드는 단순해지지만, 이 프로젝트는 이미 Nginx가 필수 컴포넌트(req/03
  명시)이므로 그 역할을 활용하는 것이 자연스럽고, Node 프로세스를 게임
  로직에 전념시켜 30Hz 틱 예산(RQ-60) 안에서 정적 파일 I/O가 이벤트
  루프를 방해할 여지를 없앤다.
- **TLS 미요구(RQ-80)에도 종단 구성을 열어두는 이유**: 사내망/로컬
  전용이라는 배포 전제가 바뀌면(예: 외부 접속 허용) TLS가 즉시 필요해질
  수 있다 — 지금 인증서 발급·강제 없이도 Nginx 설정 구조(서버 블록
  분리)만 미리 마련해두면 그 전환이 설정 변경만으로 끝난다.
- 버린 대안:
  - **bvwebchat 방식(단일 컨테이너, Node가 정적 서빙까지 겸함)**: req/03이
    Redis를 이미 지정해 그 자체로 여러 프로세스/컨테이너 구성이
    불가피하다 — 채팅 앱과 달리 이 프로젝트는 상태 저장소가 프로세스
    내부에 머물 수 없어 단일 컨테이너 패턴을 그대로 가져올 수 없다.
  - **PaaS(관리형 배포, 예: Fly.io/Railway)**: req/03이 Docker+Nginx를
    이미 명시한다 — 대안 검토는 필수가 아니지만, RQ-80 확정(사내망/로컬
    전용)과도 근본적으로 맞지 않는다(PaaS는 공인 인터넷 노출을 전제).
  - **Redis 없이 서버 프로세스 메모리로 세션 관리**: req/03이 Redis를
    명시적으로 지정했으므로 채택 불가 — 다만 서버를 다중 인스턴스로
    스케일할 계획이 생기면 Redis가 그 확장의 전제 조건(세션 상태의
    프로세스 외부화)이 되므로, 이 선택은 미래 확장에도 유리하다.
  - **지금 TLS 종단을 기본 활성화**: RQ-80이 사내망/로컬 전용을 확정해
    TLS를 요구하지 않는다고 명시했으므로, 쓰지 않을 인증서 발급·관리
    비용을 지금 들일 이유가 없다 — 구성만 열어두는 절충을 택했다.

## 결과

- 배포 단위가 컨테이너 3개(Nginx, 게임 서버, Redis) + 볼륨 1개(SQLite)로,
  bvwebchat의 "이미지 하나"보다 운영 구성 요소가 많다 — docker-compose
  (또는 동등 매니페스트) 파일 자체가 배포 계약의 일부가 되어야 하며,
  아직 작성되지 않았다(구현 단계 산출물).
- **무중단 재배포가 확정적으로 없다**: RQ-80이 "무중단 배포는 목표가
  아니다"라고 명시했다 — "재배포 시 전원 끊김"은 이제 수용된 트레이드
  오프가 아니라 요구사항 자체다. **이 결정에서 가장 후회할 가능성이
  높은 지점**: 서비스가 커져 "재배포=전원 튕김"이 실제 문제가 되면(즉
  RQ-04의 "상시 접속"과 이 결정이 충돌하는 지점에 도달하면), Colyseus
  room 이관(seamless handoff) 같은 훨씬 복잡한 메커니즘을 나중에 얹어야
  하고, 그때는 RQ-80의 개정이 먼저 필요하다.
- **RQ-80 확정에 따라 TLS 미요구가 요구사항 자체가 됐다**(선택이 아니라
  확정) — 사내망/로컬 전용 배포엔 충분하지만, 배포 대상이 인터넷 공개로
  바뀌면(전제 변경) TLS 종단 활성화가 필수가 된다. Nginx 설정을 미리
  분리해뒀으므로 그 전환은 설정 변경만으로 가능하지만, 전환을 자동으로
  강제하는 장치는 없다 — 사람이 판단해 전환해야 한다.
- 🟡 확인 필요: SQLite 파일의 백업 정책(볼륨 스냅샷 주기 등)은 req에
  없다 — 운영 단계에서 결정한다.
