# ADR-0004: 물리·충돌 — Rapier, 서버 권위 + 클라이언트 예측 사본, kinematic 캐릭터 컨트롤러

- 상태: 승인 (근거: RQ-92 확정 — 2026-07-21 Deep Interview)
- 날짜: 2026-07-21
- 관련 스펙: RQ-18, RQ-20, RQ-21, RQ-22, RQ-32, RQ-92

## 맥락

req/03이 물리 엔진으로 Rapier를 지정했으나, 어디서(서버만? 서버+클라이언트?)
어떤 캐릭터 모델(kinematic 캐릭터 컨트롤러 vs 완전한 동역학 rigid body)로
돌릴지, 그리고 실제 이동 파라미터가 무엇인지는 RQ-92가 확정되기 전까지
정할 수 없었다. Deep Interview(질문 5~9)로 RQ-92가 확정되어 이 ADR을
"제안"에서 "승인"으로 전환한다.

## 결정

- 물리 시뮬레이션의 권위는 **서버**에 있다(RQ-61과 정합) — 서버가 Rapier로
  캐릭터 이동·충돌·낙하 판정(RQ-18)·사다리(RQ-21)·박스 점프(RQ-22)를 매 틱
  계산한다.
- 플레이어 캐릭터는 **kinematic 캐릭터 컨트롤러**로 모델링한다 — Rapier의
  KinematicCharacterController(캡슐 콜라이더 + 슬라이드/스텝 오프셋)를
  사용하고, 완전한 동역학 rigid body(토크·충돌 반작용력)는 쓰지 않는다.
- 클라이언트는 **동일한 Rapier**(WASM 빌드)를 서버와 같은 파라미터로 로컬
  실행해 예측(RQ-62)의 결정성을 맞춘다 — 클라이언트 물리는 예측용 사본이며
  진실이 아니다(서버 스냅샷 도착 시 항상 승리).
- **캐릭터 컨트롤러 파라미터(RQ-92 확정값)**:

  | 항목 | 값 |
  |---|---|
  | 기본 이동 속도 | 6 m/s |
  | 앉기 배율 | 50% (3 m/s) |
  | 천천히 걷기 배율 | 70% (4.2 m/s) |
  | 점프 높이 | 1.0 m |
  | 낙하 데미지 | 3m 초과분 1m당 10, 즉사 임계 없음 |
  | 공중 가속 | 미허용(에어 스트레이프·버니합 없음) — 공중에서는 점프
    이함 시점의 수평 관성만 유지, 방향 전환은 착지 후에만 반영 |

  슬로프 한계·스텝 오프셋 등 Rapier 콜라이더 세부 튜닝값과 박스 클러스터
  (RQ-32)의 정확한 박스 치수는 req가 수치를 정하지 않았다 — 코드가 아닌
  설정 파일의 조정 가능한 값으로 둔다(값 발명 금지). 사격 관련 파라미터
  (탄퍼짐 콘 반경 등)는 이 ADR의 범위 밖이다([ADR-0005](0005-hit-registration.md)).

## 근거

- kinematic 캐릭터 컨트롤러는 FPS 이동(RQ-20 걷기/달리기/점프/앉기/천천히
  걷기, RQ-22 박스 점프)의 표준적 선택 — 동역학 rigid body는 다른 물체와
  부딪힐 때 원치 않는 회전·미끄러짐이 생겨 예측 가능한 게임 필(feel)을
  만들기 어렵다(Source/Quake 계열도 캡슐+kinematic 방식).
- 서버·클라이언트 동일 물리 엔진 채택 이유: 예측(RQ-62)이 서버와 다른
  결과를 내면 매 프레임 정정(snap correction)이 발생해 미끌거림이
  심해진다. 같은 엔진·같은 연산 순서를 쓰면 예측 오차가 순수 네트워크
  지연(입력 처리 시점 차이)만으로 좁혀진다.
- 공중 가속 미허용(RQ-92)은 kinematic 캐릭터 컨트롤러의 공중 상태 전이
  로직을 단순화한다 — 착지 전까지 수평 속도가 상수이므로 별도의 공중
  조작 입력 처리 분기가 필요 없다. 예측 정확도에 미치는 구체적 영향은
  결과 절 참고.
- 버린 대안:
  - **동역학 rigid body 캐릭터**: 물리 반응(임펄스·토크)으로 인한 비의도적
    거동(경사에서 미끄러짐, 충돌 시 튕김)을 통제하기 어렵고 FPS 조작감과
    맞지 않는다.
  - **서버만 Rapier, 클라이언트는 자체 간이 이동 코드로 예측**: 구현은
    단순하지만 서버·클라이언트 충돌/낙하 판정 로직이 두 벌로 갈라져
    예측 오차가 구조적으로 커진다 — 특히 RQ-22 박스 점프처럼 충돌 형상에
    민감한 동작에서 예측이 자주 빗나간다.
  - **물리 엔진 없이 자체 AABB/캡슐 이동 코드**: req/03이 Rapier를 이미
    지정했고, 사다리(RQ-21)·박스 점프(RQ-22) 같은 볼륨 기반 특수 이동을
    신뢰성 있게 구현하려면 검증된 충돌 엔진이 필요하다.
  - **공중 가속 허용(에어 스트레이프)**: CS류 이동 숙련도를 살릴 수
    있으나, 인터뷰(질문 8)에서 넷코드 결정론을 우선해 명시적으로
    기각했다 — 아래 결과 참고.

## 결과

- 클라이언트 번들에 Rapier WASM이 포함돼 초기 로드 크기가 커진다 —
  RQ-01(설치 없이 브라우저 실행)과 초기 로딩 시간 사이의 트레이드오프.
- 서버·클라이언트 Rapier 버전과 설정(중력·타임스텝·충돌 마진)을 엄격히
  동기화해야 한다 — 버전 드리프트가 예측 오차의 조용한 원인이 될 수 있다.
  후회 지점: 의존성 업데이트 시 두 곳을 항상 같이 올려야 하는 결합이 생긴다.
- 사다리(RQ-21)·박스 점프(RQ-22)는 Rapier 센서 콜라이더로 구현한다 —
  사다리 2개·박스 클러스터 3곳(RQ-32)의 구체 배치는 [ADR-0007](0007-map-asset-pipeline.md)의
  명명된 볼륨과 맞물린다.
- **공중 가속 미허용이 캐릭터 컨트롤러 설계·예측 정확도에 주는 영향
  (RQ-92 확정)**: 캐릭터 컨트롤러는 공중 상태에서 입력에 반응하는 별도
  분기(에어 스트레이프 가감속)를 구현할 필요가 없다 — 점프 이함 시점의
  수평 속도 벡터를 고정 상수로 유지하다가 착지 시에만 재평가한다. 이는
  서버와 클라이언트가 동일한 궤적을 계산하는 데 필요한 상태가 "점프 시점
  속도 + 경과 시간 + 중력"뿐이라는 뜻이며, 예측 재생([ADR-0003](0003-netcode-authority.md))이
  입력 시퀀스의 미세한 타이밍 차이에 영향받지 않고 결정론적으로
  수렴한다. 반대로 에어 스트레이프를 허용했다면 공중에서도 매 틱 입력을
  반영해야 하므로 클라이언트-서버 간 부동소수 연산 순서 차이가 누적돼
  예측 오차(및 정정 스냅)가 늘었을 것이다 — CS류 이동 숙련도(공중 조작)
  라는 게임성 축을 포기한 대가로 넷코드 신뢰성을 얻은 명시적
  트레이드오프다(질문 8 근거).
- **잔존 미해결 사항(승인 상태와 무관)**: 서버(Node.js) 환경에서 Rapier
  headless 실행이 30Hz 틱 예산(33.3ms) 안에서 10인 규모 연산을 충분히
  빠르게 처리하는지는 아직 실측하지 않았다. RQ-92 확정으로 파라미터가
  정해져 이 ADR을 승인으로 전환하지만, 이 성능 가정은 별도로 검증돼야
  한다 — 실측에서 예산 초과가 확인되면 이 ADR의 재검토 대상이다.
