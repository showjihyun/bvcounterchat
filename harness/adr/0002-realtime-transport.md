# ADR-0002: 실시간 전송 계층 — Colyseus(WebSocket) on Node.js + Fastify

- 상태: 승인 (근거: docs/req/03)
- 날짜: 2026-07-21
- 관련 스펙: RQ-04, RQ-40, RQ-60, RQ-61, RQ-80

## 맥락

상시 접속 단일 상설 세션(RQ-04), 전원 대상 채팅 브로드캐스트(RQ-40), 서버
30Hz 고정 틱(RQ-60), 서버 권위 상태 동기화(RQ-61)를 지원하는 실시간 양방향
전송이 필요하다. req/03이 서버 스택(Node.js + Fastify + Colyseus/WebSocket)을
이미 확정했다. Deep Interview(질문 27·28)로 RQ-80이 배포 대상을 **사내망/
로컬 전용**으로, RQ-64가 **RTT 150ms 이내 정상 플레이 보장**을 확정했다 —
이는 아래 결과의 TCP HOL 리스크 평가가 성립하는 전제 조건이 된다.

## 결정

**Colyseus**를 룸 기반 실시간 상태 동기화 엔진으로 채택하고, **Fastify**를
HTTP 서버(정적 자산·헬스체크·보조 REST 엔드포인트)로 같은 프로세스에 붙인다.
트랜스포트는 WebSocket 단일 경로.

## 근거

- Colyseus의 Room + Schema(필드 단위 바이너리 델타 동기화) 모델이
  RQ-60/61(서버가 매 틱 상태를 브로드캐스트, 클라이언트는 이를 신뢰하되
  그대로 진실로 확정하지 않음)을 프레임워크 수준에서 구조화한다. 직접
  구현하면 델타 인코딩·룸 생명주기·재연결까지 처음부터 만들어야 한다.
- 룸 개념이 RQ-03(10인 정원)·RQ-04(상시 단일 세션)와 1:1 대응 — 서버 기동
  시 생성되는 단일 영구 룸 인스턴스로 모델링한다.
- Schema 동기화는 필드 단위 델타만 전송해 대역폭을 줄인다 — 10명 규모에서도
  위치·회전·HP·탄약 등 고빈도 필드가 다수다.
- 버린 대안:
  - **순수 ws(raw WebSocket)**: 룸 생명주기·상태 델타 인코딩·재연결·직렬화
    프로토콜을 전부 직접 구현해야 한다 — 이 규모 개발 리소스로는 과도한
    인프라 코드. req/03이 이미 Colyseus를 지정했으므로 채택하지 않는다.
  - **Socket.IO**: 이벤트 기반 메시징에는 강하지만 상태 동기화(스냅샷 델타)를
    위한 1급 지원이 없다. 채팅 전용 서비스(bvwebchat)에는 적합했으나,
    고빈도 위치/HP 동기화가 핵심인 FPS에는 룸+스키마 동기화 프레임워크가
    더 적합하다 — 직접 델타 로직을 얹으면 결국 Colyseus를 재발명하는 셈이다.
  - **geckos.io(WebRTC DataChannel 기반 UDP 유사 전송)**: 신뢰성 없는
    전송으로 헤드오브라인 블로킹을 피할 수 있어 FPS에 이론적으로 더
    적합하지만, (1) 시그널링 서버가 추가 구성 요소로 필요하고, (2) 브라우저
    WebRTC의 NAT/방화벽 통과 이슈가 배포 구성([ADR-0009](0009-deployment.md) Docker+
    Nginx)의 단순성을 깨며, (3) Colyseus만큼 성숙한 룸/재연결 생태계가
    없다. 10인·사내망 배포(RQ-80 확정)에서는 TCP HOL 비용을 감내 가능하다고
    판단했다(아래 결과 참고).

## 결과

- **TCP 헤드오브라인 블로킹(HOL) 트레이드오프**: WebSocket은 TCP 위에서
  동작한다 — 패킷 하나가 유실/지연되면 그 뒤의 모든 메시지가 스트림 순서상
  대기한다. FPS의 위치/사격 스냅샷처럼 "최신 것만 중요하고 유실된 이전
  프레임은 버려도 되는" 데이터에는 신뢰성 없는 전송(UDP류)이 이론적으로
  우월하다. RQ-80이 배포 대상을 **사내망/로컬 전용**으로 확정하고 RQ-64가
  **RTT 150ms 이내 정상 플레이**를 보장 범위로 못박았으므로, 이 트레이드오프의
  실질 영향은 낮은 손실률·짧은 RTT라는 확정된 네트워크 전제 위에서 제한적이다.
- **이 프로젝트에서 가장 후회할 가능성이 높은 결정 — 조건부로 수용**: 사격
  판정([ADR-0005](0005-hit-registration.md))과 이동 스냅샷([ADR-0003](0003-netcode-authority.md))이
  전부 이 WebSocket/TCP 트랜스포트 위에서 동작한다 — 패킷 유실 후 재전송
  대기가 "순간 끊김"으로 체감될 수 있다. **이 리스크는 RQ-80의 사내망 전제가
  유지되는 동안에만 억제된다.** 배포 대상이 인터넷 공개(외부 접속 허용,
  공인 도메인 등)로 전환되면 이 ADR은 재검토 대상이다 — 그 시점에는
  트랜스포트 계층 교체(geckos.io 등 UDP류) 또는 최소한 패킷 손실 시뮬레이션
  기반 재검증이 필요하다. 전제가 바뀌면 되살아나는 리스크라는 것이 요점이다.
- Fastify는 Colyseus의 WebSocket 업그레이드 경로와 별도로 정적 서빙/
  헬스체크만 담당한다 — 두 서버 프레임워크가 한 프로세스·한 포트에
  공존해야 하므로 부착 방식(HTTP 서버 인스턴스 공유)을 구현 단계에서
  확정한다.
- RQ-40 채팅도 별도 서버 없이 같은 Colyseus 룸의 메시지 채널로 얹는다 —
  순서 보장은 단일 룸·단일 프로세스 이벤트 루프 전제에 기댄다(bvwebchat
  ADR-0001과 동일 패턴).
- **버전 제약 (2026-07-22 RQ-04 착수 시 확정)**: 클라이언트 `colyseus.js`와
  서버 `colyseus`/`@colyseus/schema`는 **schema 메이저가 일치해야 한다** —
  Schema의 바이너리 델타 포맷이 메이저 간 호환되지 않는다. 스캐폴딩(PR #1)이
  클라이언트 없이 서버만 0.17(schema 4)로 깔아 이 불일치가 잠복했다.
  현재 스택은 **0.16 라인으로 정렬**했다: colyseus 0.16.x + @colyseus/schema 3.x
  + colyseus.js 0.16.x (셋 다 schema 3). 클라이언트 SDK가 서버 메이저를
  따라오지 못하는 것이 Colyseus의 구조적 제약이므로, 서버 버전을 올릴 때는
  반드시 대응 클라이언트 존재를 먼저 확인한다.
