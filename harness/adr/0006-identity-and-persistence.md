# ADR-0006: 신원·영속성 — 닉네임 식별 + 익명 UUID 통계 키 + Redis 세션 + SQLite 통계

- 상태: 승인 (근거: docs/req/03, RQ-02, RQ-81 확정)
- 날짜: 2026-07-21
- 관련 스펙: RQ-02, RQ-03, RQ-41, RQ-43, RQ-81, RQ-04, RQ-91, RQ-93

## 맥락

req/03이 저장소로 Redis(세션)와 SQLite(통계)를 지정했다. RQ-02는 닉네임만
으로 식별(계정 없음)을, RQ-81은 재시작 시 세션 소실 허용·통계 보존을
요구한다. RQ-03/RQ-41은 정원 초과 시 관전자 입장과 승격을, RQ-43은 AFK
자동 퇴장과 빈 슬롯의 관전자 개방을 요구한다. Deep Interview로 RQ-03이
관전자 상한(10명, 합계 20 연결)과 정원 초과 시 거부를, RQ-81이 통계
식별자를 **닉네임이 아닌 클라이언트 보관 익명 UUID**로, RQ-91이 관전
모드 세부(자동 입장·자유 시점 기본+1인칭 추적·선착순 자동 승격)를,
RQ-93이 통계 집계 범위(서버 가동 이후 누적, 리셋 없음)를 확정했다 — 이
ADR의 미결 항목이 모두 해소됐다.

## 결정

1. **신원**: 닉네임만으로 식별(RQ-02, 계정·비밀번호 없음). 중복 닉네임은
   서버가 자동 접미사를 붙여 고유화한다. 세션 식별자는 서버가 발급해
   Redis에 보관하되, RQ-81(재시작 시 세션 소실 허용)에 따라 "재시작=새
   신원"을 기본으로 하고 짧은 재연결 유예만 동일 신원 복원을 지원한다.
2. **세션 상태(Redis)**: 접속 중 플레이어/관전자 목록, 각 플레이어의 소속
   (단일 상설 세션이므로 항상 하나), 관전자 승격 대기열(RQ-03/RQ-41,
   FIFO), AFK 타이머(RQ-43)를 Redis에 보관한다. **정원(RQ-03 확정)**:
   플레이어 최대 10명 + 관전자 최대 10명(합계 20 연결) — 관전자 정원까지
   찬 상태에서 접속이 시도되면 서버는 접속을 거부하고 사유를 응답한다.
   Redis는 "지금 누가 접속해 있는가"의 진실 공급원이며, 서버 재시작 시
   소실을 허용한다(RQ-81) — 재시작 후 전원 재접속으로 복원한다.
3. **통계(SQLite)**: 누적 킬·데스·헤드샷·플레이타임(RQ-81) 등 영구 집계는
   SQLite에 기록해 서버 재시작 후에도 보존한다. 집계 범위는 **서버 가동
   이후 누적**이며 자동 리셋은 없다(RQ-93). 게임 상태(위치·HP 등 고빈도
   값)는 SQLite에 쓰지 않는다 — 인메모리(Colyseus room state,
   [ADR-0002](0002-realtime-transport.md))가 진실이고, SQLite는 "사망 시 킬
   카운트를 1 늘린다"류의 저빈도 이벤트 로그다.
4. **통계 식별자(익명 UUID, RQ-81 확정)**: 통계 행의 키는 세션 토큰(1번)과
   **별개**로, 클라이언트가 최초 접속 시 생성해 **브라우저 localStorage에
   보관하는 익명 UUID**다. 접속 시 클라이언트가 이 UUID를 서버에 제시하고,
   서버는 검증 없이 그대로 통계 행 키로 사용한다(서버가 발급하지 않는다).
   닉네임(RQ-02)은 화면 표시용일 뿐 통계의 키가 될 수 없다.
5. **재연결 유예**: 연결 종료 시 즉시 퇴장 처리하지 않고 짧은 유예 시간을
   두어 새로고침·일시 단절을 흡수한다. 유예 내 재연결 시 동일 신원(닉네임,
   HP, 탄약, 킬 수)을 Redis에서 복원한다. 유예 만료 시 퇴장 확정 → 슬롯이
   관전자 대기열의 맨 앞 인원에게 열린다(RQ-03/RQ-41/RQ-43).
6. **관전자 승격 대기열**: FIFO 큐로 Redis에 유지 — 정원 초과로 입장한
   관전자와 순수 관전 선택 관전자를 구분하지 않고 도착 순서로 큐잉한다.
   플레이어 자리가 나면 큐의 최선착순 관전자를 자동 승격한다(RQ-91 확정).
   관전 카메라(자유 시점 기본, 1인칭 추적 전환)는 클라이언트 렌더링
   관심사이므로 이 ADR은 큐잉 순서만 다루고 카메라 구현은
   [ADR-0001](0001-client-rendering-stack.md)/`harness/workflow/fe.md` 영역이다.

## 근거

- Redis: 세션류 상태는 자주 갱신되고 재시작 시 버려도 되는 값(RQ-81)이라
  인메모리 캐시 성격의 스토리지가 적합하다 — SQLite에 세션을 넣으면 매
  갱신마다 디스크 I/O가 발생해 불필요한 지연이 생기고, RQ-81의 "재시작 시
  소실 허용"과도 맞지 않는 부자연스러운 선택이 된다.
- SQLite: 통계는 저빈도·영구 보존(RQ-81)이 핵심이라 파일 기반 관계형 DB로
  충분하다 — 별도 DB 서버 운영 부담 없이 단일 파일로 배포([ADR-0009](0009-deployment.md))에
  동봉할 수 있다.
- **RQ-02/RQ-81 모순 해소(질문 25 근거)**: 그대로 두면 두 요구가 모순된다
  — 닉네임은 계정처럼 안정적인 키가 아니라 매 접속 자동 고유화(RQ-02
  접미사)되는 표시 문자열이라 영구 통계의 키가 될 수 없다. 식별(가벼운
  닉네임, RQ-02 유지)과 통계 키(안정적 익명 UUID)를 분리하면 계정 시스템
  없이도 개인별 영구 통계(RQ-81)가 가능해진다.
- 버린 대안:
  - **세션도 SQLite에 저장**: RQ-81이 세션 소실을 명시적으로 허용하는데
    영속 저장소에 넣는 것은 요구를 거스르고, 상시 갱신되는 접속 목록·AFK
    타이머에 파일 DB 락 경합 위험이 생긴다.
  - **통계도 Redis에 저장**: RQ-81이 통계 보존을 요구하는데 Redis는
    기본이 인메모리다(영속 옵션을 켜더라도 req/03이 이미 SQLite를 통계
    저장소로 명시) — req가 이미 역할을 나눠 지정했으므로 재배치할 이유가
    없다.
  - **관전자 대기열을 클라이언트 로컬 상태로 관리**: 여러 관전자가 각자
    "내 순서"를 신뢰할 수 없고(클라이언트 신뢰 금지, RQ-61 원칙의 연장),
    동시 접속 경쟁 시 순서가 어긋난다 — Redis 서버측 큐로 원자적 처리.
  - **닉네임을 통계 키로 사용**: RQ-02가 자동 접미사·고유화를 요구해 동일
    인물이 재접속마다 다른 키가 될 수 있고, 접미사가 해제된 뒤 다른
    사용자가 같은 닉네임 문자열을 재사용하면 통계가 섞인다 — 채택 불가.
  - **서버 발급 계정 식별자(로그인)**: RQ-02가 계정·비밀번호 없음을
    명시 — 채택 불가.

## 결과

- 서버 재시작 시: 접속 중이던 플레이어/관전자 전원 연결 끊김, 관전자
  대기열 소실, AFK 타이머 리셋 — 전원 재접속이 필요하다(RQ-81 수용
  범위). 반면 누적 킬/데스/헤드샷/플레이타임 통계는 보존된다.
- 재연결 유예 시간(구체적 초)은 req가 정하지 않았다 — 실측·튜닝이 필요한
  조정 가능한 설정값으로 둔다.
- Redis 의존성이 배포 구성([ADR-0009](0009-deployment.md))에 별도 컨테이너/서비스로
  들어간다 — RQ-80 "단일 호스트"가 "단일 서버 프로세스"가 아니라 "단일
  호스트의 여러 컨테이너"로 해석돼야 함을 뜻한다.
- 후회 지점: 관전자 대기열이 FIFO 단일 정책이다 — "친구 우선" 등 다른
  승격 정책이 요구되면(req에 없음) Redis 자료구조를 다시 설계해야 한다.
- **기기 변경 시 별개 인격이 되는 한계**: 익명 UUID는 브라우저
  localStorage에만 존재한다 — 기기를 바꾸거나 브라우저 데이터를 지우면
  새 UUID가 발급되어 통계 이력이 끊기고 완전히 별개의 인격(새로운 0킬
  플레이어)이 된다. 계정 시스템이 없는 v1(비요구 목록, req/01)의 알려진
  한계이며, 여러 기기의 통계를 병합할 방법은 없다.
- **UUID 위조 가능성의 수용**: 서버는 클라이언트가 제시하는 UUID를
  검증하지 않고 그대로 신뢰한다 — 클라이언트가 UUID를 조작해 타인의
  누적 통계에 기록을 얹거나, 자신의 통계를 초기화(새 UUID 생성)할 수
  있다. 안티치트가 v1 범위 밖(비요구 목록, RQ-61이 최소 방어선)이라는
  결정과 일관되게, 이 위조 가능성은 v1에서 명시적으로 수용한다.
- RQ-91(관전 모드)·RQ-93(통계 집계 범위)이 확정되어 이 ADR의 남은 미결
  항목이 모두 해소됐다 — SQLite 스키마는 리셋 로직을 두지 않는다(RQ-93
  "리셋 없음").
