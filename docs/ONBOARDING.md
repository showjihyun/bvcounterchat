# ChatStrike 온보딩 — 첫 주 가이드

> 대상: 팀에 새로 합류한 개발자.
> 목표: **첫 PR을 머지하기까지** 필요한 것만 담는다. 나머지는 링크로 넘긴다.
> 분량은 15분 안팎 읽기. 환경 세팅 시간은 네트워크에 달렸다(실측 안 함).

> **이 문서는 다른 문서의 요약이다.** 내용이 어긋나면 언제나 원본이 이긴다 —
> 규칙·컨벤션은 `CLAUDE.md`, 문서 지도는 `harness/README.md`, 상태 기호는
> `harness/progress.md`가 정본이다. 이 문서는 진실 공급원 서열에 없다.

이 프로젝트는 일반적인 웹 프로젝트와 **작업 흐름이 다르다.** 규칙이 몇 개
더 있고, 그 규칙을 사람이 아니라 hook·CI·AI 에이전트가 강제한다. 낯설 수
있어서 각 규칙에 **왜 있는지**를 같이 적었다. 이유를 모르는 규칙은 지켜지지
않고, 지켜지지 않는 규칙은 없는 것과 같다.

---

## 1. 이 프로젝트

채팅이 중심인 웹 기반 3D FPS. 브라우저만으로 실행되고, 최대 10명이
개인전(FFA)으로 붙는다. 로그인은 없고 닉네임만 쓴다.

| | |
|---|---|
| 클라이언트 | React + React Three Fiber + Three.js + Zustand + Vite |
| 서버 | Node.js + Fastify + Colyseus (WebSocket) |
| 물리 | Rapier · 30Hz 고정 틱 · 서버 권위 |
| 저장소 | Redis(세션) + SQLite(통계) |

원본 요구사항은 `docs/req/`에 있고, 그걸 정규화한 것이 
`harness/specs/requirements.md`다. **후자가 진실 공급원이다.**

## 2. 첫날 — 환경 세팅

```bash
git clone https://github.com/showjihyun/bvcounterchat.git
cd bvcounterchat
npm ci
npm run check      # 이게 통과하면 준비 끝
```

`npm run check`는 이 순서로 돈다:

```
스펙 동결 게이트 → lint → typecheck → test
```

실행해 볼 것:

```bash
npm run dev          # 클라이언트 (http://localhost:5173)
npm run dev:server   # 서버 (http://localhost:2567/health)
npm test             # 테스트만  (test:watch 도 있다)
npm run typecheck    # 타입만
npm run gate         # 스펙 동결 상태만 확인
npm run build        # 클라이언트 + 서버 빌드
```

**커밋·브랜치 컨벤션**

```
브랜치   <타입>/<설명>               예: docs/onboarding, chore/tdd-pipeline
         RQ 구현이면 feat/<RQ-ID>-<설명>   예: feat/RQ-04-persistent-room
커밋     <타입>(<범위>): <설명>      타입: feat | fix | chore | test | docs
```

테스트 커밋은 `test(RQ-04): ...`, 구현 커밋은 `feat(RQ-04): ...`로 나눈다 —
순서가 지표 M3의 측정 근거라서 한 커밋에 합치면 측정이 안 된다.

## 3. 30초 요약 — 일이 흘러가는 방식

```
  스펙 확인            없으면 만들지 않는다 (RQ가 있어야 시작)
      ↓
  원장에 기록          harness/progress.md 에 행 추가 + 🔄
      ↓
  브랜치               feat/<RQ-ID>-<설명>
      ↓
  테스트 먼저 (Red)    실패하는 테스트 + 테스트 커밋
      ↓
  구현 (Green)         테스트를 통과시키는 최소 구현
      ↓
  독립 평가            제3자 세션이 검증 → PASS / FAIL / BLOCKED
      ↓
  PR                   스펙·ADR 변경이 있으면 같은 PR에
      ↓
  이중 게이트          CI(기계적) + reviewer(추론적) 둘 다 통과
      ↓
  머지
```

빨간 단계(테스트 먼저 / 구현 / 독립 평가)는 **각각 격리된 AI 에이전트 세션**이
맡는다. 왜 그런지는 6절에.

## 4. 여섯 가지 규칙과 그 이유

### ① 스펙에 없으면 만들지 않는다

`harness/specs/requirements.md`에 RQ로 없는 기능은 구현하지 않는다.
좋은 아이디어라도 먼저 스펙에 추가하고 시작한다.

**왜**: 스코프 크리프를 막기 위해서가 아니라, **검증 기준이 없는 코드를 막기
위해서**다. RQ가 없으면 "이게 맞게 동작하는가"를 판단할 근거가 없다.

**안 지키면**: reviewer가 스코프 이탈로 blocker 판정 → 머지 차단.

### ② 테스트가 먼저다

구현 커밋보다 테스트 커밋이 앞서야 한다.

**왜**: 나중에 쓴 테스트는 구현에 맞춰진다. 이미 만든 것을 통과시키는 테스트는
검증이 아니라 기록이다.

**안 지키면**: **자동으로 잡히지 않는다.** 커밋 순서를 검사하는 게이트는 없다 —
지표 M3(테스트 선행률)를 주간 회고 때 `git log`로 사후 집계할 뿐이다. 지금은
규율로만 지켜진다. (CI의 "테스트-코드 동행 검사"는 순서가 아니라 같은 PR에
테스트가 함께 바뀌었는지만 보며, 경고일 뿐 머지를 막지 않는다.)

### ③ 문서는 코드와 같은 PR에서 바뀐다

스펙·ADR을 고쳐야 하는 변경이면 그 수정을 **같은 PR에** 넣는다.

**왜**: 따로 하면 반드시 한쪽이 뒤처진다. 뒤처진 문서는 다음 사람을 틀린
방향으로 보낸다 — 그리고 그 사람은 문서를 믿을 이유가 있었다.

**예외**: 아직 코드가 없는 RQ의 신설·개정은 코드 없이 할 수 있다
(이 시점을 **구현 게이트**라 하고, `CLAUDE.md`가 유일한 정의처다).

### ④ 게이트를 우회하지 않는다

CI 실패, reviewer의 REQUEST_CHANGES, 스펙 동결 게이트 — 급하다는 이유로
넘기지 않는다.

**왜**: 한 번 우회하면 그 게이트는 그날로 장식이 된다. 다음부터는 "저번에도
그냥 갔잖아"가 된다.

### ⑤ 확인하지 않은 것을 확인했다고 하지 않는다

"통과했습니다"라고 쓰려면 실행 출력이 있어야 한다. 못 본 것은 "미확인"으로
남긴다.

**왜**: 이 프로젝트의 게이트는 대부분 보고서 기반이다. 보고가 부정확해지는
순간 게이트 전체가 거짓 신호를 낸다.

### ⑥ 내가 만든 것을 내가 판정하지 않는다

테스트는 **직접 돌린다**(그건 의무다). 하지만 "이 변경이 합격인가"를 판정하는
것은 **격리된 세션**이 한다 — evaluator와 reviewer.

증거를 **만드는** 것과 그 증거로 합격을 **판정**하는 것은 다른 일이다.
앞은 작업자가, 뒤는 제3자가 한다.

**왜**: 자기 작업은 의도대로 읽힌다. 내가 방금 쓴 문장은 나에게 늘 말이 된다.
실제로 이 규칙을 헌법에 넣은 직후, **그 규칙 자체가 헌법의 다른 줄과 모순되는
것을 격리 리뷰어가 잡았다.** 작성자였던 세션은 세 번 읽고도 못 봤다
(`harness/changelog.md` 2026-07-21 항목).

**안 지키면**: 자기 채점이 되고, 게이트가 통과 신호만 내는 장식이 된다.

## 6. 왜 AI 에이전트가 각 단계를 맡나

핵심은 **격리**다. 같은 세션이 테스트도 쓰고 구현도 하면 구현에 맞춘 테스트가
되고, 평가자가 구현자의 설명을 들으면 자기 채점이 된다.

| 단계 | 담당 | 제약 | 무엇이 강제하나 |
|---|---|---|---|
| Red | test-writer | 구현을 쓰지 않는다 | 프롬프트 |
| Green | coder | 테스트를 고치지 않는다 | 프롬프트 + evaluator 사후 검출 |
| 평가 | evaluator | 구현자의 설명을 받지 않는다 | 입력이 파일뿐 — **구조적** |
| 머지 리뷰 | reviewer | 작성자의 의도 설명을 받지 않는다 | 입력이 diff뿐 — **구조적** |

앞 두 줄은 **도구 권한으로 막히지 않는다.** coder는 `Edit`·`Bash`를 갖고 있어
테스트 파일을 고칠 수 있다. `tests/` 쓰기 차단 hook은 미착수다
(`harness/progress.md` 17f). 뒤 두 줄만 구조로 보장된다.

데이터는 `_workspace/{RQ-ID}/` 파일로만 오간다. 대화는 넘기지 않는다.

**이게 실제로 값을 한다.** 리뷰 게이트는 첫 가동에서 lint 규칙의 구멍과
"재리뷰가 조용히 1회차로 처리되는" 결함을 잡았고, 두 번째 PR에서는 평가
검사 하나가 **항상 빈손으로 통과하는** 상태였음을 검출해 머지를 막았다.

### 왜 사람이 아니라 AI가 승인하나

이 게이트는 **솔로 체제**에서 만들어졌다. GitHub은 자기 PR을 자기가 승인할 수
없으므로, 사람 리뷰어의 자리를 격리된 에이전트가 대신한다.

**팀에 사람이 늘면 이 전제가 바뀐다.** 그때는 ADR로 역할을 재정의한다 —
사람 리뷰를 필요조건으로 올리고 에이전트를 선행 필터로 둘지, 아니면 현행을
유지할지. 지금은 **현행 유지**이며, reviewer의 APPROVE가 머지의 필요조건이다.

> ⚠️ 다만 GitHub 브랜치 보호가 아직 설정되지 않았다. 지금 이 규칙은
> **규율로만** 지켜진다 — 마음먹으면 그냥 머지할 수 있다. 하지 마라.

## 7. 작업 한 건의 전체 흐름

각 단계 옆 괄호는 **실제로 일어난 예**다 — RQ-04(상설 세션, PR #5)의 흔적.

1. **원장 확인** — `harness/progress.md`에서 할 일을 찾거나 행을 추가하고 🔄
   (RQ-04: 원장 19a 행)
2. **스펙 확인** — RQ가 ✅인가? 🟡면 인터뷰가 먼저다 (게이트가 막는다).
   매핑된 GA 케이스가 없으면 여기서 멈추고 **사람이 정답을 쓴다**
   (RQ-04: GA 케이스 0건으로 멈춤 → 사용자가 GA-26~29 결정, 커밋 `45233f7`)
3. **브랜치** — `feat/<RQ-ID>-<설명>` (RQ 구현이 아니면 `<타입>/<설명>`)
4. **파이프라인 실행** — `tdd-workflow` 스킬. Red → Green → 평가
   (RQ-04: 테스트 커밋 `25de339` → 구현 커밋 `a8389d2` — 이 순서가 M3의 증거)
5. **PR** — 검증 출력을 본문에 넣는다 (RQ-04: PR #5 본문 참조)
6. **리뷰 게이트** — `review-gate` 스킬. APPROVE 없이 머지 금지
   (RQ-04: 평가 3회 + 리뷰 2회를 거쳤다 — 왜 그랬는지는 10절)
7. **머지 후** — 원장 ✅, 골든 케이스 `status` 갱신(사람 승인 하에)
   (RQ-04: 머지 커밋 `da5a2f1`, GA-26~29 done)

## 8. 자주 막히는 곳

| 증상 | 원인 | 대응 |
|---|---|---|
| `src/` 수정이 차단됨 | 스펙에 🟡가 남아 있다 | `npm run gate`로 확인 → 인터뷰 먼저 |
| **파이프라인이 Phase 0에서 멈춤** | 그 RQ에 매핑된 골든 케이스가 없다 | **정상이다.** RQ 44건 중 GA 케이스가 있는 건 19건뿐이라 자주 만난다. 케이스를 사람이 먼저 써야 한다 — 에이전트가 자기 정답을 쓰게 하지 않는다 |
| 테스트가 Red가 아니라 에러로 죽음 | 테스트 파일 자체의 타입 오류 | 고친다. 단 아직 없는 `src/` 모듈 임포트(TS2307/2305)는 정당한 Red |
| 로컬(Windows)만 실패 | 콘솔 인코딩 — cp949 stdout에 em-dash 출력 | **실제로 한 번 있었다.** 게이트가 스펙과 무관하게 `UnicodeEncodeError`로 죽었다 → `_force_utf8()`로 대응(`gate_spec_freeze.py`) |
| CI(Linux)만 실패하고 로컬은 통과 | OS 차이 — `Path`가 POSIX에서 `\`를 경로 구분자로 보지 않는다 | **실제로 한 번 있었다**(커밋 `06cb0a2` — selftest가 CI에서만 실패). 로컬 재현은 WSL·컨테이너에서. 게이트는 `\`→`/` 정규화로 대응했다 |
| reviewer가 REQUEST_CHANGES | blocker가 있다 | 고치고 재리뷰. 우회하지 않는다 |
| 원장에 뭘 써야 할지 모르겠음 | — | 상태 기호는 ⬜ 대기 · 🔄 진행중 · ✅ 완료 · ⛔ 차단(사유 병기) |

## 10. 워크드 예제 — RQ-04가 실제로 흘러간 길

첫 제품 RQ(상설 세션, **PR #5**)의 전체 궤적이다. 매끄럽지 않았다 —
그리고 그 매끄럽지 않음이 이 파이프라인이 존재하는 이유다.

| 단계 | 무슨 일이 있었나 | 커밋 |
|---|---|---|
| Phase 0 | GA 케이스 0건으로 **멈춤** → 사용자가 검증 기준 결정(GA-26~29). Colyseus 클라·서버 schema 메이저 불일치도 여기서 발견 — 코드 쓰기 전에 | `45233f7` |
| Red | test-writer가 블랙박스 통합 5건. 룸 클래스를 임포트하지 않고 실 포트에 접속 — 구현 위치가 바뀌어도 테스트가 안 흔들린다 | `25de339` |
| Green | coder 구현. Fastify `onClose` 훅에서 WebSocket 교착을 만나 `preClose`로 해결 — 실 네트워크 테스트가 아니면 못 봤을 버그 | `a8389d2` |
| 평가 1차 | **FAIL.** evaluator가 요구받은 5회를 넘어 **102회** 반복해 ~2% 콜드스타트 워커 크래시 검출 | — |

### 이 한 건이 보여주는 것

- **정답은 사람이 쓴다.** Phase 0이 멈춘 건 고장이 아니라 설계다 — 검증
  기준 없이 구현하면 "맞게 동작하는가"를 판단할 근거가 없다.
- **반복 횟수가 검출력이다.** 30회 스모크는 ~2% flaky를 절반 이상(0.98³⁰≈55%)의
  확률로 놓친다.
  evaluator는 102회·270회를 돌렸고, 그때마다 "고쳐졌다"는 확신을 뒤집었다.
- **작성자의 확신은 증거가 아니다.** 오케스트레이터도 두 번 틀렸고, 두 번
  다 격리 세션이 잡았다. 규칙 ⑥이 사람에게만 적용되는 게 아니다.
- **게이트마다 보는 게 다르다.** evaluator는 "계약을 충족하는가"에서 flaky를,
  reviewer는 "머지해도 되는가"에서 재시도 설계의 사각지대를 잡았다. 하나로
  합쳤으면 둘 중 하나는 통과했다.
